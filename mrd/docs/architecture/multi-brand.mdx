# Multi-Brand & Multi-Tenant Architecture

## Overview

The MEDrecord ecosystem uses a **Brand + Tenant** model where:
- **Brand** = Shared application codebase (HealthTalk, Coachi, MedSafe)
- **Tenant** = Healthcare institution with its own Medplum Project (complete data isolation)

This ensures NEN 7510 / AVG compliance with full data separation between healthcare institutions.

### Hierarchy

| Level | Medplum Concept | MEDrecord Usage | Example |
|-------|-----------------|-----------------|---------|
| **Brand** | App codebase | Shared Next.js app with branding | HealthTalk, Coachi, MedSafe |
| **Tenant** | **Project** | Healthcare institution - complete data isolation | Ziekenhuis Amsterdam, Huisarts Rotterdam |
| **User** | ProjectMembership | User within a tenant | Dr. Smith at Ziekenhuis Amsterdam |

### Why Tenant = Project (Not Organization)

| Concern | Organization-based | Project-based (Our Choice) |
|---------|-------------------|---------------------------|
| **Data Isolation** | Soft (AccessPolicy) | Hard (native Medplum) |
| **NEN 7510 Compliance** | Risk - shared DB | Compliant - separate data |
| **Breach Containment** | All tenants exposed | Only 1 tenant affected |
| **Audit Logs** | Mixed per-tenant | Clean per-tenant |
| **Scaling** | Single project limits | Independent scaling |

### Gateway Multi-Tenant Integration

The Gateway provides the `tenant_id` (Medplum Project ID) on every authenticated request:

```typescript
// Gateway response includes tenant context
{
  "auth": {
    "user_id": "user-uuid",
    "tenant_id": "healthtalk-ziekenhuis-a",  // Medplum Project ID
    "brand": "healthtalk"
  }
}
```

All API routes MUST verify `tenant_id` from Gateway and scope queries accordingly.

```json
  ]

```

## Medplum Integration

### Project-Level Isolation (Brands)

Each brand operates within its own Medplum Project. This provides:

- **Complete data isolation** between brands
- **Independent configuration** (AccessPolicies, Bots, Subscriptions)
- **Separate FHIR endpoints** per brand

### Organization-Level Multi-Tenancy (Within Brands)

Within each brand, Organizations represent tenants (clinics, hospitals, departments). This uses Medplum's native multi-tenant access control.

## Environment Configuration

Each tenant deployment requires specific environment variables:

```bash
# Brand identification (determines UI theme)
NEXT_PUBLIC_BRAND=healthtalk
NEXT_PUBLIC_BRAND_NAME="HealthTalk"

# Gateway (shared across all tenants)
GATEWAY_URL=https://auth-test-b2c.healthtalk.ai

# Tenant-specific Medplum Project
MEDPLUM_PROJECT_ID=healthtalk-ziekenhuis-a    # Tenant ID from Gateway
MEDPLUM_CLIENT_ID=xxx                          # Per-tenant client
MEDPLUM_CLIENT_SECRET=xxx                      # Server-side only
```

### Deployment Model

Each tenant gets its own Vercel deployment of the brand app:

| Tenant | Domain | Medplum Project | Brand |
|--------|--------|-----------------|-------|
| Ziekenhuis A | `ziekenhuis-a.healthtalk.ai` | `healthtalk-ziekenhuis-a` | healthtalk |
| Huisarts B | `huisarts-b.healthtalk.ai` | `healthtalk-huisarts-b` | healthtalk |
| Praktijk X | `praktijk-x.coachi.ai` | `coachi-praktijk-x` | coachi |

Alternative: Single deployment with runtime tenant resolution from Gateway.

## Security Requirements

All brand applications MUST follow these security patterns:

| Requirement | Implementation |
|-------------|----------------|
| Server-side only backend operations | Use Next.js Server Components, Server Actions, Route Handlers |
| No direct client-to-Medplum calls | All requests go through Gateway |
| Tenant-scoped queries | Every request verified with `auth.id` from Gateway |
| No secrets in client code | Environment variables without `NEXT_PUBLIC_` prefix |

## Creating a New Brand

To create a new brand application:

1. **Copy the template** from an existing brand in `mrd/apps/` folder
2. **Update package.json** with new brand name (`@mrd/newbrand`)
3. **Create brand theme** in `mrd/packages/ui/src/themes/newbrand.css`
4. **Customize `.agents/`** folder with brand-specific MDX overrides

## Onboarding a New Tenant

To onboard a new healthcare institution:

1. **Create Medplum Project** for the tenant (e.g., `healthtalk-ziekenhuis-nieuw`)
2. **Configure Project.setting** with tenant-specific branding overrides
3. **Set up AccessPolicies** for the tenant's user roles
4. **Create Vercel deployment** with tenant-specific env vars (or configure runtime resolution)
5. **Configure Gateway** to recognize the new `tenant_id`
6. **Copy template Questionnaires** from brand template project if needed

### Project.setting for Runtime Branding

Brand configuration can be stored in the Medplum Project.setting for runtime theming:

| Setting Key | Description | Example |
|-------------|-------------|---------|
| `brand.id` | Brand identifier | `coachi` |
| `brand.name` | Display name | `Coachi` |
| `brand.primaryColor` | Primary color (OKLCH) | `oklch(0.55 0.18 145)` |
| `brand.accentColor` | Accent color | `oklch(0.92 0.05 145)` |
| `brand.logoUrl` | Logo URL | `https://...` |
| `brand.features` | Enabled features (comma-separated) | `chat,video,scheduling` |

Use `getProjectTheme()` from `@mrd/ui` to load these settings at runtime.

See [Agent Configuration](./agent-configuration.mdx) for details on customizing agent behavior per brand.
