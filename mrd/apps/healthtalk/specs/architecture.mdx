## FHIR Version Landscape

There is no "FHIR STU4". The correct version sequence is: **STU3 вҶ’ R4 вҶ’ R4B вҶ’ R5 вҶ’ R6**.

| FHIR version | Dutch ZIB release | Status for HealthTalk |
|--------------|-------------------|----------------------|
| STU3 | zib2017 | Legacy вҖ” phased out by Nictiz. Do not use for new builds. |
| **R4** | **zib2020 (`nl-core`)** | Implement now |
| **R4** | **zib2024 (`nl-core`)** | Published Oct 2024 вҖ” use for new `Questionnaire` resources |
| R5 | вҖ” | Skip вҖ” official HL7 NL advice |
| R6 | вҖ” | Future only |

**Active package:** `nictiz.fhir.nl.r4.nl-core` (pin to `0.12.0-beta.1`)  
**Track for migration:** `nictiz.fhir.nl.r4.zib2024`

> The MedMij `QuestionnaireProvisioningTask` profile was formally published in STU3. For R4, HealthTalk uses the `nl-core-Task` base profile. When Nictiz publishes the formal R4 questionnaire IG, update the profile URL only вҖ” no structural changes required.

## Key Architectural Principles

### Gateway-First Delivery

All channel integrations live **behind a Notification Gateway**, not in the HealthTalk Vercel frontend. HealthTalk (Vercel) calls a single internal endpoint. The gateway owns:

- Every provider credential
- Every delivery adapter
- All retry logic
- All fallback routing
- All status normalisation back to MedPlum

```
HealthTalk (Vercel)          Notification Gateway              Providers
      в”Ӯ                      gateway.healthtalk.ai
      в”Ӯ  POST /notify        в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
      в”Ӯ  { patientId,        в”Ӯ  Channel selector    в”Ӯ
      в”Ӯ    taskId,           в”Ӯ  Per-org config      в”Ӯв”Җв”Җв–ә WhatsApp (Twilio)
      в”Ӯ    channel: 'auto' } в”Ӯ  Retry + fallback    в”Ӯв”Җв”Җв–ә Email (Postmark)
      в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–әв”Ӯ  Status normaliser   в”Ӯв”Җв”Җв–ә SMS (Twilio)
                             в”Ӯ  Audit вҶ’ MedPlum     в”Ӯв”Җв”Җв–ә Voice mobile (ElevenLabs+Twilio)
                             в”Ӯ                      в”Ӯв”Җв”Җв–ә Voice SIP (ElevenLabs+SBC)
                             в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

      HealthTalk never imports Twilio, ElevenLabs, or Postmark SDKs.
      All provider webhooks land at the gateway, not at Vercel.
```

**Benefits:**

- HealthTalk frontend is not coupled to any provider
- New channels are added to the gateway вҖ” zero frontend changes
- All credentials in one place with one security boundary
- Retry, fallback, and audit logic written once
- Each adapter is independently testable
- Provider can be swapped without touching HealthTalk

**Deployment:** The gateway is a separate Vercel project (`gateway.healthtalk.ai`), same infrastructure, completely isolated. Its own env vars, its own domain, its own deploy pipeline.

## Multi-Tenant Model

HealthTalk uses a **Tenant = Medplum Project** model for complete data isolation between healthcare institutions.

### Tenant Resolution

The Gateway provides `tenant_id` on every authenticated request:

```typescript
// Gateway auth response
{
  "auth": {
    "user_id": "practitioner-uuid",
    "tenant_id": "healthtalk-ziekenhuis-a",  // Medplum Project ID
    "brand": "healthtalk",
    "roles": ["practitioner", "admin"]
  }
}
```

### API Route Pattern

Every API route MUST verify tenant context from Gateway:

```typescript
// app/api/tasks/route.ts
export async function GET(request: Request) {
  const auth = await verifyGatewayAuth(request);
  
  // tenant_id IS the Medplum Project - all queries are automatically scoped
  const medplum = getMedplumClient(auth.tenant_id);
  
  const tasks = await medplum.searchResources('Task', {
    status: 'requested',
    // No need for tenant filter - Project isolation handles it
  });
  
  return Response.json(tasks);
}
```

### Deployment Options

| Model | Description | Use Case |
|-------|-------------|----------|
| **Per-tenant deployment** | Each tenant has own Vercel deployment | Large institutions, custom domains |
| **Shared deployment** | Single deployment, runtime tenant resolution | Smaller tenants, cost efficiency |

See [Multi-Brand Architecture](/mrd/docs/architecture/multi-brand.mdx) for full details.

### Task-First Lifecycle

A `QuestionnaireProvisioningTask` must be created before anything else happens. It is the single source of truth for the lifecycle of every questionnaire assignment. The `QuestionnaireResponse` is an output of the Task, not a standalone object.

- The Task exists even if delivery fails
- The clinician dashboard is a Task queue, not a QR list
- Reminders, escalations, and cancellations operate on the Task
- MedPlum Subscriptions fire on Task status changes
