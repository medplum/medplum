# Binary

A resource that represents the data of a single raw artifact as digital content accessible in its native format. A <Link to="/docs/api/fhir/resources/binary">Binary</Link> resource can contain any content, whether text, image, pdf, zip archive, etc.

  <TabItem value="usage" label="Usage">
<div> <a name="scope"></a> <p> There are situations where it is useful or required to handle pure binary content using the same framework as other  resources. Typically, this is when the binary content is referred to from other FHIR Resources.  Using the same framework means that the existing servers, security arrangements, code libraries, etc.  can handle additional related content. Typically, <Link to="/docs/api/fhir/resources/binary">Binary</Link> resources are used for handling content such as: </p> <ul>  <li><a href="http://www.hl7.org/implement/standards/product_brief.cfm?product_id=7">CDA </a> Documents (i.e. with XDS)</li>  <li>PDF Documents</li>  <li>Images</li> </ul> <p> A binary resource can contain any content, whether text, image, pdf, zip archive, etc. These  resources are served in their native form on the rest interface, but can also be represented in XML, JSON, or other formats,  such as when including these resources in a <Link to="/docs/api/fhir/resources/bundle">Bundle</Link> (used when it is convenient to include these in the  response directly rather than leaving them by reference). </p> </div>

  <TabItem value="relationships" label="Relationships">
<div> <a name="bnr"></a> <p> When a FHIR server finds it convenient to manage the content within the same overall REST framework  as the other resources, the <Link to="/docs/api/fhir/resources/binary">Binary</Link> resource is generally used as the target in the  <a href="https://www.hl7.org/fhir//api/fhir/datatypes/attachment">Attachment</a> data type to reference content via the <code>.url</code>  element, such as in the <Link to="/docs/api/fhir/resources/documentreference">DocumentReference</Link> and <Link to="/docs/api/fhir/resources/media">Media</Link> resources. Consequently, the <Link to="/docs/api/fhir/resources/binary">Binary</Link> resource can  be a target wherever the <Link to="/docs/api/fhir/datatypes/attachment">Attachment</Link> data type is used such as in the <Link to="/docs/api/fhir/resources/documentreference">DocumentReference</Link> resource.  </p> <p> The <a href="/docs/api/fhir/resources/documentreference">DocumentReference</a> and <a href="/docs/api/fhir/resources/media">Media</a> resources  allow conveying binary content (via attachment) or pointing to one (as a <code>Binary</code> or non-FHIR  URI) along with the metadata around that resource, and as such are searchable.   <Link to="/docs/api/fhir/resources/binary">Binary</Link> resources do not support 'search'. </p> <p> While CDA and PDF documents are conveyed as <Link to="/docs/api/fhir/resources/binary">Binary</Link> (because they cannot be expressed natively in FHIR),  <a href="https://www.hl7.org/fhir/documents.html">FHIR Documents</a> do not need to be similarly encoded and can be sent natively  in FHIR using <a href="/docs/api/fhir/resources/bundle">Bundle</a>.  However, in some situations FHIR Documents may be sent  as a <Link to="/docs/api/fhir/resources/binary">Binary</Link> if there is a need to treat them the same as other types of documents or binary files. </p> <p> The <Link to="/docs/api/fhir/resources/binary">Binary</Link> resource does not convey context of the file. If the context (information such as  author, procedure, technique, etc.) should be conveyed, <Link to="/docs/api/fhir/resources/media">Media</Link> or <Link to="/docs/api/fhir/resources/documentreference">DocumentReference</Link> resources are  appropriate. The <Link to="/docs/api/fhir/resources/binary">Binary</Link> resource may be used to convey actual binary file content conveyed by  those resources. </p> <p> The <a href="/docs/api/fhir/resources/media">Media</a> resource is preferred for handling images, but this is not possible when the content is already binary (e.g. in some uses of <a href="https://wiki.ihe.net/index.php/Cross-Enterprise_Document_Sharing">IHE XDS </a>). </p> </div>
