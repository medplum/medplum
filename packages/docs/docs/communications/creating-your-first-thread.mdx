# Creating Your First Thread

This walkthrough creates a thread through the FHIR API using [`MedplumClient`](/docs/sdk/core.medplumclient). That matches how a real app (or Bot) writes messages. The [Medplum App](https://app.medplum.com) is useful as an inspector: confirm resources, references, and history—not as the primary authoring experience you ship to users.

Before you start, read how thread headers and messages differ in [Organizing Communications — Building and Structuring Threads](/docs/communications/organizing-communications#building-and-structuring-threads).

The example below is a provider–patient conversation: the patient is `subject` and is included in `recipient` on the thread header and on messages where they should see or receive the thread, alongside the clinician.

## Prerequisites

- An authenticated `MedplumClient` (see [MedplumClient](/docs/sdk/core.medplumclient) and [Auth](/docs/auth)). If you still need a Medplum project, use [Register](/docs/tutorials/register).
- Existing [`Patient`](/docs/api/fhir/resources/patient) and [`Practitioner`](/docs/api/fhir/resources/practitioner) resources in that project whose `id` values match the snippet references (or change every `reference` in the code to match your data). Create or import them in the [Medplum App](https://app.medplum.com) or another flow you use for test data. Any reference that does not resolve causes `createResource` to fail with a server error pointing at the bad reference.

:::caution Creating threads in the Medplum App

You can create or edit [`Communication`](/docs/api/fhir/resources/communication) resources in the [Medplum App](https://app.medplum.com) if you are still wiring up a client, exploring the thread model, or seeding a small amount of test data. The App talks to the same FHIR API; this is a temporary or internal shortcut, not the authoring experience you ship to end users.

Manual entry is easy to get wrong: the thread header must have no `payload` and no `partOf`, while every message must include both `payload` and `partOf` pointing at the header. Set the same `topic` on the header and on every child message. On the header, list everyone who should see the thread in `recipient` (for this tutorial, the patient and the clinician who started the thread) so inbox-style queries work (see [Organizing Communications](/docs/communications/organizing-communications#building-and-structuring-threads)). [Access policies](/docs/access/access-policies) still control what each login can read or write. Prefer the API walkthrough below for anything beyond quick experiments.

:::

## Create the Thread Header and Messages

1. Thread header — A [`Communication`](/docs/api/fhir/resources/communication) with no `payload` and no `partOf`. It groups the conversation and carries `topic`, `subject`, participants, and optional `category`.
2. Each message — A `Communication` with `payload` (content) and `partOf` pointing at the same thread header. Use the same `topic` text on the header and every child, and repeat the same `category` values on each message so category-based searches match. This walkthrough sets `subject` only on the thread header; add `subject` on child messages too if your searches or profiles require it.

## Atomic Creates With Transaction Bundles

The steps above use separate `createResource` calls so each part of the thread is obvious. In production, you may prefer a single FHIR `transaction` `Bundle` so the thread header and first message are created together (all entries succeed or none are committed). Use `fullUrl` and `urn:uuid` references so the first message’s `partOf` can point at the header inside the same bundle. See [FHIR Batch Requests](/docs/fhir-datastore/fhir-batch-requests) for `transaction` bundles, the `executeBatch` helper on [`MedplumClient`](/docs/sdk/core.medplumclient), project feature flags, and current limitations.
